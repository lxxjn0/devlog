처음 설치한 **RDBMS 소프트웨어**는 데이터베이스가 비어있는 상태이다. 여기에 **테이블이나 뷰, 인덱스, 프로시저 등의 데이터베이스 객체**를 작성해 데이터베이스를 구축해나간다.

## 1. 데이터베이스 객체

**데이터베이스 객체(이하 객체)**란 테이블이나 뷰, 인덱스 등 데이터베이스 내에 정의하는 모든 것을 일컫는 말이다.

객체는 **데이터베이스 내에서 실체를 가지는 어떤 것**인데 객체의 종류에 따라 데이터베이스에 **저장되는 내용도 달라진다.**

예를 들어 우리가 지금까지 다뤘던 객체인 테이블의 경우는 행과 열이 저장된다.

> 데이터베이스를 조작하는 SQL 명령은 객체가 아니다. 데이터베이스 내부에 존재하는 것이 아니기 때문이다.

그리고 객체는 **이름**을 가진다.

데이터베이스 내에서 객체를 작성할 때는 이름이 겹치지 않도록 한다.

이름을 정할 때는 몇 가지 **제약 사항(명명 규칙)**을 따른다.

- 기존 이름이나 예약어와 중복하지 않는다.
- 숫자로 시작할 수 있다.
- 언더스코어(_) 이외의 기호는 사용할 수 없다.
- 한글을 사용할 때는 더블쿼트(MySQL에서는 백쿼트)로 둘러싼다.
- 시스템이 허용하는 길이를 초과하지 않는다.

객체의 이름은 어떤 데이터가 저장되어 있는지 파악하는 기준이 되는 경우가 많으므로 **의미있는 이름을 붙여야 한다.**

> 이는 여타 프로그래밍 언어에서의 명명과 같은 맥락인 것 같다.
>
> 항상 이름은 의미를 가지고 의도를 드러내도록 작성해야 하는 것!

한편 **이름은 객체의 종류와는 관계없이 동일한 이름을 작성할 수 없다.**

foo라는 이름의 테이블을 만들면 같은 이름을 가지는 테이블이나 뷰와 같은 객체를 만들 수 없다.

## 2. 스키마

데이터베이스 객체는 **스키마라는 그릇 안에 만들어진다.** 따라서 **객체의 이름이 같아도 스키마가 서로 다르면 상관없다.**

이와 같은 특징 때문에 **데이터베이스 객체는 스키마 객체**라고 불리기도 한다. 스키마 객체는 **SQL 명령의 DDL을 이용하여 정의**한다.

> 실제로 데이터베이스에 테이블을 작성해서 구축해나가는 작업을 **스키마 설계**라고 한다.

어떤 것이 스키마가 되는지는 데이터베이스 제품에 따라 다르다. 지금 사용하는 MySQL에서는 `CREATE DATABASE` 명령으로 작성한 데이터베이스가 스키마가 된다.

테이블과 스키마는 무엇을 담는 그릇과 비슷한데 각각의 그릇 안에서는 동일한 이름을 사용할 수 없다.

이처럼 이름이 충돌하지 않도록 기능하는 그릇을 **네임스페이스(*namespace*)**라고 부르기도 한다.

## 정리하면

지금까지 데이터베이스를 구성하는 **데이터베이스 객체**에 대해 알아봤다.

객체의 **이름**을 어떻게 설정해야 하는지, 이름은 **어떤 범위를 기준**으로 중복되면 안되는지, 또 어떤 객체들이 존재하고 어디에 포함되는지를 다시 한 번 확인해보는 계기였다.
